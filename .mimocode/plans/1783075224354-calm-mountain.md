## PRIORIDADE 1 — CORRECOES CRITICAS

### 1.1 Garantir que Scheduler funciona (verificacao + fix)

**O que ja existe:** Scheduler em `app/workers/scheduler.py` (daemon 30s), campo `scheduled_for` no Post, picker no frontend.

**Risco:** Coluna `scheduled_for` pode nao existir em DBs antigos. O `_run_legacy_migrations()` em `app/database.py:126-147` nao a adiciona.

**Acoes:**
- Adicionar migration da coluna `scheduled_for` no `_run_legacy_migrations()` de `app/database.py`
- Testar fluxo completo: selecionar post -> setar AGENDADO -> escolher data/hora -> salvar -> aguardar scheduler publicar
- Verificar que o comparador de strings ISO funciona (ja validado por pesquisa: funciona porque ISO sort == cronologico)

**Arquivos:** `app/database.py`

### 1.2 Dashboard de status na aba Posts

**O que ja existe:** Endpoint `GET /api/posts/stats/summary` retorna contadores por status. Frontend tem `loadQueueStats()` mas mostra apenas na aba Settings.

**Acoes:**
- Mover/criar dashboard de status no topo da aba Posts (index.html)
- Mostrar cards: Pendentes, Agendados, Publicados, Erros
- Adicionar acoes rapidas: Publicar agora, Reagendar, Excluir, Atualizar fila
- Reutilizar endpoint existente `GET /api/posts/stats/summary`

**Arquivos:** `templates/index.html`, `static/js/app.js`

## PRIORIDADE 2 — GERADOR DE CONTEUDO

### 2.1 Simplificar legenda (unificar campos)

**O que ja existe:** Campos separados: titulo, CTA, hashtags, caption. Backend retorna JSON com todos eles.

**Objetivo:** Legenda final unica = titulo + texto persuasivo + CTA + hashtags. Campos internos mantidos para IA.

**Acoes:**
- Na aba Conteudo, esconder campos titulo/CTA/hashtags do usuario (manter no JSON interno)
- Exibir apenas: palavras-chave, link afiliado, legenda final (textarea maior)
- A IA continua gerando titulo/CTA/hashtags internamente
- Botao "Copiar legenda" copia a legenda completa
- Ao salvar, manter todos os campos no backend (content_title, content_cta, content_hashtags) — so mudar o que o usuario ve

**Arquivos:** `templates/index.html` (lines 218-283), `static/js/app.js` (lines 760-940)

### 2.2 Legendas em lote

**O que ja existe:** Geracao individual — seleciona 1 video, gera, aprova.

**Acoes:**
- Adicionar checkbox de selecao multipla na fila de conteudo
- Botao "Gerar legendas em lote" que percorre videos selecionados
- Barra de progresso global durante processamento
- Chamar `/api/content/generate-ai` sequencialmente para cada video selecionado
- Mostrar resultado/erro por video
- Usar estado existente `STATE.postQueue` e funcao existente `renderContentQueue()`

**Arquivos:** `templates/index.html`, `static/js/app.js`, possivelmente `web/routes/content.py`

### 2.3 Sugestao automatica de hashtags

**O que ja existe:** Campo `content_hashtags` no post, hashtags base nas settings.

**Acoes:**
- No prompt do Groq/Gemini, adicionar instrucao para gerar hashtags relevantes ao nicho/baseado no conteudo
- Adicionar seletor de nicho (Casa, Tecnologia, Beleza, Moda, Pets, Infantil) na aba Conteudo
- Salvar nicho como setting: `content_niche`
- Hashtags geradas aparecem no campo hashtags para edicao antes de aprovar
- Usar o nicho no prompt do LLM para hashtags mais direcionadas

**Arquivos:** `app/groq_content.py`, `app/gemini_content.py`, `templates/index.html`, `static/js/app.js`

## PRIORIDADE 3 — EDITOR

### 3.1 Thumbnail na fila de conteudo

**O que ja existe:** `renderContentQueue()` mostra apenas filename + status badge. Backend tem endpoint `POST /api/editor/thumbnail` que extrai frame via FFmpeg.

**Acoes:**
- Ao renderizar fila de conteudo, extrair/guardar thumbnail para cada video
- Adicionar miniatura ao lado do nome na lista
- Usar endpoint existente `POST /api/editor/thumbnail` ou extrair frame direto no frontend via canvas
- Mostrar nome + duracao + badge de status

**Arquivos:** `templates/index.html`, `static/js/app.js`, `static/css/app.css`

### 3.2 Frame de capa

**O que ja existe:** FFmpeg extrae frames para thumbnails. Video processor tem acesso ao FFmpeg.

**Acoes:**
- Adicionar checkbox "Gerar frame de capa" na aba Editor
- Ao processar, gerar frame inicial do video (00:00:01) como imagem separada
- Salvar como `{nome_video}_capa.jpg` na pasta saida/
- Adicionar log informando onde a capa foi salva
- Opcao de usar como thumbnail do post automaticamente

**Arquivos:** `app/video_processor.py`, `web/routes/editor.py`, `templates/index.html`, `static/js/app.js`

**Acoes:**

- Permitir edicao manual dos horarios gerados
- Salvar configuracao: `smart_schedule_posts_per_day`, `smart_schedule_times`

**Arquivos:** `app/workers/scheduler.py`, `web/routes/posts.py`, `templates/index.html`, `static/js/app.js`

## REGRAS DE IMPLEMENTACAO

- Ler codigo existente antes de criar funcoes novas
- Reutilizar `renderContentQueue()`, `renderPostQueue()`, `loadQueueStats()`
- Reutilizar endpoints existentes (`/api/posts/stats/summary`, `/api/editor/thumbnail`)
- Nao duplicar logica — se existe, estender
- Cache busting `?v=N` ao alterar JS/CSS
- Status em ingles/uppercase: AGENDADO, PROCESSANDO, PUBLICADO, ERRO
- IDs sao UUID hex sem hifens
- Todo commit push no GitHub
